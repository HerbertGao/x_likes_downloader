## 上下文

当前 `organize_files.rs` 的 `organize_files(a_dir, b_dir)` 方法通过扫描目标目录 `b_dir` 中的文件夹名构建前缀映射表，再将源目录 `a_dir` 中的文件按解析出的 username 进行一对一匹配。同一个人拥有多个 X 账号时，只有与文件夹名前缀完全匹配的账号文件能被归档，其余全部跳过。

## 目标 / 非目标

**目标：**
- 支持将多个 X username 映射到同一个目标文件夹
- 映射关系通过 `{DOWNLOAD_DIR}/username_aliases.txt` 文本文件定义
- 映射文件不存在时静默忽略，保持完全向后兼容
- 移动后的文件保留原始 username，不改写文件名

**非目标：**
- 不引入 YAML/TOML 等结构化配置格式（避免新依赖）
- 不修改 `.env` 配置体系
- 不改变现有的文件夹前缀匹配逻辑
- 不支持正则或通配符匹配

## 决策

### 决策 1：别名文件格式 — 简单文本，每行一组逗号分隔

```
alice, alice_art, alice_photo
bob, bob_backup
```

每行第一个 username 为主名称，用于匹配目标文件夹。后续 username 为别名。

- **替代方案 A**：YAML 格式 — 更结构化，但需引入 `serde_yaml` 依赖，项目当前无此依赖
- **替代方案 B**：JSON 格式 — 已有 `serde_json`，但对用户手动编辑不友好
- **选择理由**：纯文本最简单，用户可直接用任何编辑器修改，零依赖

### 决策 2：别名文件位置 — `{DOWNLOAD_DIR}/username_aliases.txt`

通过 `organize_files` 的 `a_dir` 参数（即 DOWNLOAD_DIR）拼接路径，不引入额外配置项。

- **替代方案**：在 `.env` 中增加 `ALIAS_FILE` 配置项指定路径
- **选择理由**：约定优于配置，减少用户配置负担。文件放在下载目录中，用户操作时触手可及

### 决策 3：别名查询插入点 — 在现有匹配逻辑之前

```
解析文件名 → 得到 username
     ↓
查别名表（HashMap<String, String>: alias → primary）
     ↓ 有映射则替换，无则保留
用（可能已替换的）username 执行现有匹配逻辑
```

- **选择理由**：最小侵入，不修改现有匹配逻辑，仅在其上游增加一层转换

### 决策 4：别名解析容错

- 忽略空行和 `#` 开头的注释行
- 每个 username 去除首尾空格
- 同一 alias 出现在多行时，后者覆盖前者（last-write-wins）
- 格式错误的行（如只有一个 username）静默跳过

## 风险 / 权衡

- **用户不知道此功能** → 在 README 和 `env.example` 中添加说明
- **别名文件格式错误** → 容错解析，格式错误的行跳过而非报错，不影响正常归档
- **同一别名映射到多个主名称** → last-write-wins 策略简单但可能导致非预期行为。考虑到文件很小且用户手动维护，这是可接受的权衡
